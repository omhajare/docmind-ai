Here is a comprehensive, top-to-bottom technical breakdown of everything we have built in the DocMind AI project. I have audited both the frontend and backend architectures to ensure not a single technical detail is missed.

1. High-Level Architecture
We have built a full-stack, AI-powered research assistant with two core modes: Conversational Q&A (RAG) and an Autonomous Deep Research Agent.

Backend: Python 3.11+, FastAPI, LangGraph, LangChain, PostgreSQL (via raw SQL + psycopg2), and ChromaDB.
Frontend: React 18, Vite, TypeScript, Tailwind CSS v4, and shadcn/ui components.
AI Models: Google Gemini 2.5 Flash (for LLM reasoning/generation) and Gemini Embeddings. Tavily API for web search.
2. Database & Storage Layer
We eschewed heavy ORMs like SQLAlchemy in favor of highly optimized raw SQL queries using psycopg2.extras.RealDictCursor for speed and simplicity.

Schema Implementation: We wrote a robust initialization script (database/db_client.py) that sets up pgcrypto for UUIDs and creates 10 interconnected tables:
users, refresh_tokens, password_reset_tokens for identity management.
documents and suggested_questions for file tracking.
chat_sessions and chat_messages for conversation persistence.
research_jobs for tracking background agent states.
app_settings for global configurations (like an AI kill switch).
eval_results for future LLM benchmarking.
Vector Database: ChromaDB is used to store document chunks and their semantic vectors. Collections are uniquely named per user to ensure strict tenant isolation.
File Storage: Abstracted storage service that saves raw PDFs either to local disk or Supabase Storage depending on the environment.
3. Authentication & Security System
We implemented an enterprise-grade authentication flow from scratch.

Backend Auth (jwt_handler.py, password_handler.py):
Secure password hashing using bcrypt.
Short-lived Access Tokens (JWT) and long-lived Refresh Tokens (stored in the DB for manual revocation/rotation).
Secure password reset logic utilizing hashed, expiring tokens.
Frontend Auth (AuthContext.tsx, client.ts):
Global useAuth hook providing access tokens, refresh functions, and logout state.
Advanced Axios Interceptor: Automatically catches 401 Unauthorized responses, queues any pending API requests, calls the refresh token endpoint, and replays the failed requests transparently to the user without interrupting their flow.
Security Middleware: FastAPI incorporates slowapi for Rate Limiting, standard CORS middleware, and an injected X-Content-Type-Options: nosniff header.
4. Document Management & RAG Pipeline (Mode 1)
We built a highly efficient pipeline to ingest user documents and make them queryable.

Ingestion & Validation: Uploaded PDFs are validated for MIME type and file size. We also enforce a maximum document limit per user.
Processing (rag/indexer.py):
The PDF is temporarily saved, parsed using pdfplumber to extract text.
Text is chunked, embedded using Google Gemini Embeddings, and indexed into ChromaDB.
The chunk_count and file_size are updated in the SQL database.
LLM Suggested Questions: Upon successful upload, we trigger an LLM prompt that reads the first few chunks of the document to auto-generate 3 contextually relevant "suggested questions" for the user to click on.
Cleanup: Fully integrated cascade deletes. Deleting a document removes the raw file from storage, the embeddings from ChromaDB, and the database record simultaneously.
5. Conversational Q&A Engine (Chat Service)
The core chat experience goes beyond simple RAG by incorporating tool use and confidence scoring.

State Management: Chats are organized into Sessions. The system automatically titles the session based on the user's first query. Message history is serialized and kept in PostgreSQL.
The RAG Flow (services/chat_service.py):
The user's query is embedded, and ChromaDB is queried for the highest matching chunks.
The system checks the cosine_score of the retrieved chunks.
Conditional Web Search Fallback: If the cosine score is low (meaning the local document doesn't hold the answer), the system falls back to querying the Tavily API for real-time web data.
Both Document Context and Web Context are injected into a highly formatted system prompt alongside recent conversation history.
Metadata Tagging: Every message saves metadata about what sources were used (PDF filenames + page numbers, or Web URLs) and assigns a Confidence Score (High, Medium, Low) based on retrieval accuracy.
6. Autonomous Deep Research Agent (Mode 2)
The crown jewel of the project: an asynchronous, multi-step LangGraph agent that conducts deep research and writes reports.

State Machine (agent/graph.py):
Planner Node: Takes the user's topic and breaks it down into distinct sub-questions.
Researcher Node: Searches both RAG and Web for each sub-question concurrently.
Reflector Node: Analyzes the findings against the original topic to identify "Coverage Gaps" (what did we miss?).
Gap Filler Node: A conditional loop that explicitly searches for the identified gaps. (Loops until gaps are filled or max iterations hit).
Synthesizer & Writer Nodes: Organizes all findings and drafts the final text.
Background Execution: The agent runs in a FastAPI BackgroundTask. We implemented state tracking (queued, researching, reflecting, etc.) that the frontend can poll to show real-time progress.
DOCX Generation (report/docx_writer.py):
The system compiles the LangGraph output into a heavily formatted Word Document using python-docx.
It creates 7 sections: A stylized Title Page (with colored confidence scores), Executive Summary, Sub-Questions, Findings, Coverage Gaps, Synthesis, and formatted References (both local PDFs and Web URLs).
7. Frontend User Interface
Modern Routing: Fully setup react-router-dom with lazy-loaded components (Research Page, Admin Page) and Suspense fallbacks to keep initial bundle sizes tiny.
Theming & Components: Utilized shadcn/ui alongside next-themes and Tailwind CSS to create a beautiful, accessible UI.
Pages Built: Landing Page, robust Auth flows (Login, Signup, Forgot/Reset Password), Dashboard (document uploads and chat interface), and the Deep Research interface.
API Services Layer: Clean extraction of API calls into auth.api.ts, chat.api.ts, documents.api.ts, and research.api.ts.
8. DevOps & Resiliency
Stale Job Recovery: Built a startup event in main.py that scans the database for stuck background research jobs and marks them as failed so they don't clog the UI.
Global Kill Switch: A database-driven app_settings toggle allowing an admin to disable LLM calls globally if costs spike or APIs go down, returning graceful 503 Service Unavailable errors.

14. Repository Layer Pattern (5 repositories)

Clean separation: user_repository, token_repository, document_repository, chat_repository, research_repository — each encapsulating raw SQL for a single domain entity.
15. Token Security — SHA-256 Hashing

Refresh tokens and password reset tokens are never stored in plaintext. The token_repository hashes them with SHA-256 before DB storage and looks them up by hash.
16. Refresh Token Rotation

On every /auth/refresh call, the old token is revoked and a new one is issued — preventing token reuse attacks.
17. Anti-Enumeration on Forgot Password

forgot_password() always returns the same success message regardless of whether the email exists — preventing attackers from probing for valid accounts.
18. Password Reset Revokes All Sessions

After a password reset, revoke_all_user_tokens() invalidates every refresh token for that user, force-logging out all devices.
19. Research Job Cancellation Checks

Every agent node (planner, researcher, reflector, gap_filler, writer) checks job.status == "cancelled" before proceeding — enabling mid-execution cancellation.
20. Research Report Retention & Cleanup

Jobs have an expires_at column (default 7 days). On startup, cleanup_expired_jobs() deletes expired DB rows and their DOCX files from disk.
21. Cookie Security — Environment-Aware

Production: httpOnly=True, secure=True, samesite=none.

22. Silent Session Restore on Page Refresh

AuthContext runs refresh() on mount, decodes the JWT payload client-side via atob() to restore user info without an extra API call.

DocMind AI — RAGAS Evaluation System Architecture
This document provides a comprehensive, top-to-bottom technical breakdown of the fully integrated RAGAS Evaluation Dashboard System in DocMind AI.

1. High-Level Architecture
The evaluation system operates as an independent micro-frontend (Streamlit) communicating securely with the core backend (FastAPI), while using LangChain and RAGAS libraries to score pipeline quality.

Components:

Streamlit Dashboard (eval_dashboard/dashboard.py): The control center for admins to manage test data, trigger evaluations, and visualize metrics.
Evaluation API Router (backend/api/v1/routers/eval.py): Secure backend endpoints for persisting and querying evaluation data.
RAGAS Runner (eval_dashboard/ragas_runner.py): The localized evaluation engine that uses Gemini 3.1 Pro as an "LLM-as-a-Judge" to score responses.
PostgreSQL Persistence (backend/repositories/eval_repository.py): Long-term storage for evaluation runs and granular per-question metrics.
React Admin Page (frontend/src/components/admin/AdminPage.tsx): A read-only summary view of historical evaluations within the main web app.
2. The "Dual-Pass" Evaluation Strategy
The most significant engineering feature of this system is its Dual-Pass Evaluation. Because DocMind AI includes an autonomous web-search fallback (Tavily), it is difficult to determine if a correct answer came from the internal ChromaDB documents or from the internet.

To isolate and evaluate just the RAG retrieval quality, the backend /chat/{session_id}/messages endpoint accepts a rag_only=true query parameter.

During evaluation, the dashboard runs every test question twice:

Pass 1 (RAG-Only): Web search is strictly disabled. This rigorously tests ChromaDB chunking, embedding quality, and cosine similarity thresholds.
Pass 2 (Full Pipeline): Web search fallback is enabled. This simulates real-world user experience and shows the "lift" provided by external search.
3. Core RAGAS Metrics
We operate the evaluation in Reference-Free mode (no human-provided "ground truth" answers are required). The RAGAS engine evaluates three critical dimensions:

Metric	What it Measures	LLM-as-a-Judge Logic
Faithfulness	Hallucination rate	Extracts all factual statements from the generated answer, then checks if every single one can be logically deduced from the retrieved context.
Answer Relevancy	Alignment to intent	Generates reverse-questions based on the generated answer. Computes cosine similarity between the reverse-questions and the original user question.
Context Utilization	Retrieval accuracy	Evaluates the retrieved chunks (or web snippets) to determine if they actually contained the necessary information to answer the question.
Note: All metrics yield a score between 0.0 and 1.0. A global EVAL_PASS_THRESHOLD (default 0.70) determines if a question "passes".

4. End-to-End Workflow Breakdown
Step 1: Automated Bot Authentication
When the admin enters the ADMIN_PASSWORD in Streamlit, the dashboard securely handshakes with the backend. It uses standard API routes to either log in or auto-register a dedicated service account: eval@docmind.ai. All subsequent test documents and chat sessions are scoped to this tenant, preventing pollution of real user data.

Step 2: Dynamic Test Data Provisioning
Instead of hardcoded data, the admin orchestrates the test environment dynamically:

Document Upload: Admin uploads test PDFs directly into the Streamlit UI. The dashboard sends these to POST /api/v1/documents/upload, indexing them into the eval bot's ChromaDB collection.
Question Generation: Admin clicks "Auto-Generate", prompting the dashboard to call GET /api/v1/documents/{id}/suggested-questions. The backend LLM reads the newly uploaded PDFs and generates targeted test questions, which are added to the queue.
Step 3: Execution & LLM Judging
When "Run Dual-Pass Evaluation" is clicked:

For each question, the dashboard creates a new API chat session.
It sends the question via POST /chat/{session_id}/messages?rag_only=true (Pass 1).
It repeats the process with rag_only=false (Pass 2).
It extracts the generated answer and the contexts (document chunks and web snippets) from the API response payload.
It formats these into SingleTurnSample objects and feeds them into ragas.evaluate().
RAGAS utilizes langchain_google_genai to spawn Gemini instances that act as impartial judges, returning a DataFrame of scores.
Step 4: Persistence & Session Cleanup
The dashboard POSTs the raw score dictionaries to the new POST /api/v1/eval/results endpoint.
The backend eval_repository.py processes the payload, calculates the boolean passed flag based on the 0.70 threshold, and executes a bulk INSERT into the eval_results PostgreSQL table.
The dashboard then immediately calls DELETE /api/v1/chat/sessions/{id} to destroy the temporary chat sessions, preventing the eval@docmind.ai account from hitting its 20-session limit.
5. Backend Infrastructure Details
Database Schema (eval_results)
sql
CREATE TABLE eval_results (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    run_id            UUID NOT NULL,       -- Groups questions from the same batch
    eval_mode         TEXT NOT NULL,       -- 'rag_only' or 'full_pipeline'
    question          TEXT NOT NULL,
    answer            TEXT,
    faithfulness      FLOAT,
    answer_relevancy  FLOAT,
    context_precision FLOAT,
    passed            BOOLEAN,             -- True if all 3 metrics >= threshold
    evaluated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- Indexed by run_id and eval_mode for fast retrieval
Secure Admin Endpoints
To prevent unauthorized users from viewing or manipulating evaluation data, all /api/v1/eval/* endpoints require an X-Admin-Password header. This is validated synchronously against the server's .env configuration before any repository methods are invoked.

6. Main Web App Integration
While Streamlit serves as the active execution environment, the main React application provides a read-only integration point.

The AdminPage.tsx component fetches data from GET /api/v1/eval/runs. This backend query uses SQL GROUP BY and AVG() functions to aggregate the raw rows into high-level run summaries. The React UI renders these as color-coded badges, giving the administrator an instant overview of historical pipeline performance (e.g., observing if a recent tweak to chunk sizes improved average Context Utilization) directly from the primary DocMind dashboard.
